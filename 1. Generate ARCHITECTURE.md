# Prompt: Generate ARCHITECTURE.md

You are a senior software architect. I have just cloned this repository and need you to produce a single file `docs/ARCHITECTURE.md` that captures the system's structure clearly enough that any future LLM session (or new engineer) can understand the project in under 10 minutes by reading only this file.

## Step 1 — Discover

Before writing anything, inspect the repository:

- Read the root manifest(s): `package.json`, `pyproject.toml`, `go.mod`, `Cargo.toml`, `pom.xml`, `composer.json`, etc.
- Read `README.md`, `Makefile`, `Dockerfile`, `docker-compose.yml`, `.env.example`.
- Scan the top 2 levels of the directory tree.
- Open entry points (`main.*`, `index.*`, `app.*`, `cmd/*`, `src/main/*`).
- Identify config files for routing, ORM, build tools, and CI.

If something is ambiguous, state your assumption inline rather than guessing silently.

## Step 2 — Write `docs/ARCHITECTURE.md` with these sections

### 1. One-paragraph summary

What this project _is_, what it _does_, and who uses it. Plain English. No marketing.

### 2. Tech stack at a glance

Bulleted: language(s) + version, framework(s), database(s), cache, queue, key libraries. Note version where it materially affects behavior.

### 3. High-level architecture

- Type of system (monolith / modular monolith / microservices / library / CLI / SPA / SSR / etc.)
- An ASCII or Mermaid `graph TD` diagram showing the major components and how requests/data flow between them.
- Trust boundaries (what's public, what's internal, what talks to a DB).

### 4. Module / directory map

A table:

| Path      | Responsibility | Key files                   | Depends on         |
| --------- | -------------- | --------------------------- | ------------------ |
| `src/api` | HTTP layer     | `router.ts`, `middleware/*` | `services`, `auth` |
| ...       | ...            | ...                         | ...                |

Only list directories that carry meaning — skip `node_modules`, `dist`, etc.

### 5. Data flow (the critical paths)

Pick the **2–4 most important user-facing flows** (e.g. "user signup", "submit order", "ingest webhook"). For each:

- Trigger
- Sequence of calls (file → file → file)
- Side effects (DB writes, queue publishes, external API calls)
- Where errors are handled

### 6. Data model

- Primary entities and their relationships (Mermaid `erDiagram` if a DB exists).
- Where the schema lives (migration files, ORM models, types).

### 7. External integrations

Every outbound dependency: third-party APIs, SaaS, message brokers, object storage. For each: what it's used for, where the client is configured, which env var holds the credential.

### 8. Cross-cutting concerns

Auth, logging, error handling, config loading, feature flags, i18n, caching — one short paragraph each, with file paths.

### 9. Build, run, deploy

- How to run locally (commands, ports).
- How tests run.
- How it ships (CI file path, container, target environment).

### 10. Known gotchas & non-obvious decisions

Anything you noticed that a newcomer would trip over: unusual patterns, legacy corners, TODO/FIXME clusters, tight coupling, or comments hinting at past pain.

### 11. Open questions

List things you could _not_ determine from the code alone — ask me directly so I can fill them in.

## Constraints

- Be concrete. Reference real file paths, not generic descriptions.
- Prefer tables and diagrams over prose where they're clearer.
- Do not invent features. If you can't verify it, say so.
- Keep the whole document under ~800 lines.
- Output the file content only — no preamble, no "here is your document" wrapper.
