# Prompt: Architecture Document

You are a senior software architect. The user has cloned this repository and wants `docs/ARCHITECTURE.md` — a single, authoritative document that captures **what the system is, how it's structured, why it's structured that way, and what a future engineer or LLM session needs to know to reason about it correctly.**

This is the deeper, more deliberate version of the document scaffolded in #1. Where #1 was about getting the structure on the page quickly so a new contributor could orient, **this version is about producing the document the team would point external auditors at, hand to a new staff engineer on day one, and use to settle architectural disagreements.** It captures the system at a level of fidelity that makes it useful for **architectural reasoning** — pre-mortems, capacity planning, security review, refactor proposals, AI-assisted modification — not just onboarding.

The deliverable is **not a tour**. It's a **reference**. A new engineer should be able to read a section to answer a specific question (e.g. "how is auth handled in the worker tier?") and find a precise, current answer with file paths.

The document must be honest about what's load-bearing, what's incidental, what's vestigial, and what the team would change if they could. A document that paints a tidy picture and skips the messy parts is worse than no document — it teaches people the wrong model and lets the messy parts rot unobserved.

## Step 1 — Discover before writing

Before producing any prose, ground every claim in the actual code. Read enough to be precise:

- **Manifests, lockfiles, runtime versions** — for every language present
- **Entry points** — `main`, `index`, `server`, `app`, `cmd/*`, `bin/*`, framework-defined entrypoints (Next.js `app/`, Django `wsgi.py`, etc.)
- **Build, run, deploy** — `Makefile`, `package.json` scripts, `Dockerfile`, `docker-compose.yml`, IaC (`terraform/`, `cdk/`, `pulumi/`, `serverless.yml`), CI workflows
- **Routing & API surface** — every place a request enters the system: HTTP routers, RPC servers, GraphQL schema, webhook receivers, queue consumers, scheduled jobs, CLI subcommands
- **Data layer** — schema files (`schema.prisma`, `models.py`, ORM models), migration history, raw SQL files, seed data
- **Cross-cutting infrastructure** — auth middleware, logging setup, error handling, config loading, feature flags, observability
- **Tests** — test directory layout, fixture conventions, what tier covers what
- **Existing docs** — `README`, `CONTRIBUTING`, `docs/`, ADRs, runbooks, RFCs. **Read these first**; if they're current, the architecture document references them rather than restating them. If they're stale, note that as a finding.
- **Configuration & secrets** — `.env.example`, config schemas, secrets manager wiring
- **Boundaries** — every place the system talks to something external: third-party SDKs, internal services it depends on, internal services that depend on it (where you can tell)

Cross-check the obvious things — versions in manifests vs lockfiles vs Dockerfile vs CI image; routes declared vs handlers existing; env vars in `.env.example` vs config schema vs actual reads. Inconsistencies are findings worth surfacing.

If the project is large enough that you can't hold all of this at once, **scope explicitly**. State which subsystems you're treating as deep (rich detail) and which as shallow (boundary references only). A precise, honest document about half the system is more useful than a vague document about all of it.

## Step 2 — Decide the document's structure for _this_ system

The structure below is a strong default — but the architecture document should match the system, not a template. Skip sections that don't apply (e.g. omit the data-flow diagrams for a pure CLI tool); add sections the system needs (e.g. "Multi-tenancy model" for a B2B SaaS; "Plugin system" for an extensible framework).

**Default sections, in order:**

1. One-paragraph summary
2. System purpose & scope
3. Quality attributes (the constraints that shaped the architecture)
4. Tech stack at a glance
5. High-level architecture
6. Module / directory map
7. Critical data flows
8. Data model
9. Concurrency & state
10. External integrations
11. Cross-cutting concerns
12. Build, deploy, runtime topology
13. Observability
14. Security model
15. Performance & scaling characteristics
16. Reliability & failure modes
17. Known gotchas, vestigial parts, and load-bearing accidents
18. Decisions & rationale (or pointer to ADRs)
19. Open questions & areas of active change

Each section, in detail:

### 1. One-paragraph summary

Three to five sentences. What the system _is_, what it _does_, who uses it, and the single most important thing to know about its shape (e.g. "monolith with a worker tier", "event-driven across three services", "static site with a small dynamic API"). Plain English. The summary should let a new reader skip the rest if all they needed was orientation.

### 2. System purpose & scope

- **Why this exists** — the problem it solves and for whom (in domain terms, not marketing).
- **In scope** — what this system is responsible for.
- **Out of scope / explicitly delegated** — what this system deliberately does not do, and what handles those responsibilities instead. This boundary is often more useful than the in-scope list.
- **Consumers** — who calls this system (internal services, external apps, end users), and through which interfaces.

### 3. Quality attributes

The properties the architecture is built to optimize for, in priority order. These are the _forces_ that shape every other section.

Examples (pick the ones that actually drove decisions in this codebase):

- Latency / throughput targets
- Availability target (best-effort, business hours, 99.9, 99.99)
- Consistency model (strong / eventual / read-your-writes / causal)
- Multi-tenancy isolation level
- Compliance posture (HIPAA, SOC 2, PCI, GDPR data residency)
- Cost ceiling (especially for early-stage products)
- Time-to-market vs long-term flexibility (a startup MVP and a 10-year-old enterprise product have different shapes for honest reasons)
- Operability (small team vs large org)

Be concrete: "p95 < 200ms for the public API; no specific target for the admin tier." Vague qualities ("scalable", "fast") aren't architecture; they're aspirations.

### 4. Tech stack at a glance

A compact reference table covering language(s), runtime, framework(s), database(s), cache, queue, search, storage, auth, observability, build, deploy, hosting. Each row pinned to a source-of-truth file. (See prompt #4 for the deeper version; here, a 10-row table is enough — the full stack lives in `docs/TECH_STACK.md` if it exists.)

### 5. High-level architecture

The most-read section. A 60-second answer to "how is this system shaped?"

Include:

- **Architectural style** — monolith, modular monolith, microservices, library, CLI, SPA, SSR, edge-rendered, event-driven, layered, hexagonal, plugin host, etc. State which one and why this style fits the quality attributes above.
- **Component diagram** — Mermaid `graph TD` or `flowchart`, showing major components and the connections between them. Include trust boundaries (what's public, what's internal, what's third-party). 8–15 nodes is the right size; if you need 40, the diagram is too zoomed-out for any reader to use.
- **Boundaries** — process boundaries, network boundaries, trust boundaries, deployment boundaries. The same component can be different things depending on which boundary you're looking at; pick the one that matters here.
- **Communication styles** — synchronous HTTP/RPC, async via queue, events via pub/sub, webhooks, batch. Each edge in the diagram has a style.

### 6. Module / directory map

A table — not prose — listing the directories that carry meaning. For each:

| Path | Responsibility | Key files | Depends on | Notes |
| ---- | -------------- | --------- | ---------- | ----- |

The "Notes" column is where you say things prose would bury: "legacy pre-rewrite, slated for removal", "owns the public API contract", "heavily tested, safe to refactor", "do not modify without consulting `OWNERS`."

Skip plumbing (`node_modules`, `dist`, `.next`, `.venv`). Include test directories only if they have non-obvious structure.

### 7. Critical data flows

Pick **2–5 of the most important flows** and walk each one end to end. Don't try to cover all flows; cover the ones that, if you understand them, you can extrapolate the rest.

For each flow:

- **Trigger** — what initiates it (HTTP request, scheduled job, webhook, user click)
- **Sequence** — the call chain, with file paths. Use a Mermaid `sequenceDiagram` for flows that cross multiple services or involve async steps; prose with file:line references is fine for simpler flows.
- **State changes** — what's written, where, in what order
- **Side effects** — events emitted, queues published, third-party calls
- **Failure handling** — what happens at each step on error, what's retried, what's reported, what the user sees
- **Where authorization happens** — every meaningful trust transition

The point of this section is that a future engineer (or LLM) trying to add to this flow can find the right files and understand the order.

### 8. Data model

- **Primary entities and relationships** — Mermaid `erDiagram` if a relational DB exists; otherwise a typed model description.
- **Source of truth** — where the schema lives (migration files, ORM model, hand-written DDL).
- **Identifier strategy** — uuid v4 / v7 / cuid / ulid / serial; consistency across entities.
- **Multi-tenancy** — how tenant isolation is enforced (column? schema? database? row-level security?).
- **Soft delete / audit / temporal data** — conventions if any.
- **Denormalizations** — explicit ones the team knows about, with the reason.
- **Constraints in code vs constraints in schema** — flag where the schema doesn't enforce something the code assumes (foreign keys missing, uniqueness enforced only at the application layer, etc.). These are operational risks worth documenting.

### 9. Concurrency & state

Often missing from architecture docs; usually critical.

- **Where state lives** — in-memory per request, in-memory per worker, in DB, in cache, in client. For each, what its lifecycle is.
- **Concurrency model** — single-threaded event loop, thread-per-request, worker pool, async, actor model, serverless cold-start-per-request.
- **What's safe to assume** — does the same request always hit the same worker? does the same user? Are there sticky sessions?
- **Race conditions handled and how** — DB transactions, optimistic locking with version columns, advisory locks, idempotency keys, distributed locks, single-writer principles.
- **Background work** — queues, scheduled jobs, cron, durable workflows. For each: trigger, processor, retry semantics, idempotency posture.

### 10. External integrations

Every outbound dependency. For each:

- **What it's used for** in domain terms
- **Where the client is configured** (file path)
- **Which env var holds credentials**
- **Failure mode** — what happens when this integration is down (request fails? feature degraded? queued for retry?)
- **Whether it's on the critical path** — does losing this integration take the system down, or just degrade it?
- **Vendor lock-in** — if any (e.g. uses Stripe-specific webhook semantics, not just card-charging)

This section is the on-call's friend during an incident.

### 11. Cross-cutting concerns

For each concern, one paragraph plus file paths:

- **Authentication** — how identity is established
- **Authorization** — how permissions are checked, where the policy lives
- **Configuration** — how config is loaded, validated, and accessed
- **Secrets management** — where they come from, how they're injected
- **Logging** — library, format, fields, where logs go
- **Error handling** — error shape, propagation rules, central handlers
- **Caching** — what's cached, where, with what invalidation
- **Rate limiting & throttling**
- **Feature flags**
- **Internationalization** if applicable

The rule: a future engineer making a change that touches any of these should be able to land here and learn the local convention without grepping.

### 12. Build, deploy, runtime topology

- **Build** — what produces the deployable artifact (container image, bundle, binary, lambda zip)
- **Deploy** — CI workflow paths; environments; what gates a deploy (tests, manual approval, canary, feature flag)
- **Runtime topology** — what actually runs in production: how many instances of what, where, behind what load balancer, with what scaling rules. Include workers, schedulers, edge functions, queues — the full inventory.
- **Environment differences** — how dev / staging / prod differ in topology, data, integrations
- **Resource expectations** — rough memory/CPU footprint, especially anywhere unusual

### 13. Observability

- **Logs** — where, in what format, retention
- **Metrics** — what's emitted, where it's collected, what's dashboarded
- **Tracing** — yes/no; if yes, the propagation mechanism
- **Alerts** — pointer to the alert definitions; ideally one paragraph on the SLOs the system tracks
- **Health checks & readiness probes** — what they actually check (a 200 from `/health` that doesn't touch the DB is a smoke alarm without batteries)

This section pairs with **Reliability & failure modes** below.

### 14. Security model

- **Trust boundaries** — restate the ones from the high-level architecture, with what's enforced at each
- **AuthN/AuthZ summary** — restate from cross-cutting, in security terms
- **Sensitive data** — what counts as sensitive in this system, where it lives, how it's protected at rest and in transit
- **Threat model assumed** — public-internet-facing? authenticated-users-only? internal-only? trusted-partners?
- **Known compensating controls** — WAF, network ACLs, IAM scopes, that the architecture relies on but the code doesn't enforce
- **Compliance scope** — if the system is in PCI / HIPAA / SOC 2 / GDPR scope, say which parts and why

This section is what an external auditor or new security engineer reads first. It must not be aspirational; it must reflect what the system actually does.

### 15. Performance & scaling characteristics

- **Where the system has been load-tested** — and to what level
- **Known bottlenecks** — where the system would fall over first under growth, based on architecture, not benchmarks
- **Hot paths** — the small set of code paths that handle the majority of traffic, by file
- **Scaling axes** — what scales horizontally vs vertically; what's stateful and pinned
- **Caching layers** and their invalidation strategy
- **Backpressure & overload behavior** — what the system does when it can't keep up (queue grows, requests time out, rate-limit kicks in, falls back to a degraded mode)

### 16. Reliability & failure modes

- **What can fail** — every component on the critical path, with its failure mode
- **What we do about it** — retries (with backoff?), circuit breakers, fallbacks, degraded modes, manual runbooks
- **Recovery** — how the system recovers from each failure (auto-restart, manual intervention, data-fix script)
- **Disaster recovery** — backup posture, restore procedure, RPO/RTO if defined
- **Known incidents** — pointer to postmortems if available; the failures the team has seen are usually the most useful information in this section

### 17. Known gotchas, vestigial parts, and load-bearing accidents

**The most valuable section in any architecture document, and the one most often missing.**

- **Gotchas** — non-obvious things that trip up new contributors. The naming convention that looks wrong but matters; the file that looks unused but is loaded by glob; the env var that silently changes behavior; the migration that has to be run by hand.
- **Vestigial parts** — code or infrastructure that exists for historical reasons and would be removed in a clean rewrite. Naming them prevents people from treating them as load-bearing and replicating them.
- **Load-bearing accidents** — things that work because of an undocumented coincidence (a race window that never closes because the dependency happens to be slower, a config value that's correct only because nobody's redeployed since 2022). These are not fixed; they're noted, so when they break, the team knows where to look.
- **Anti-patterns the team is aware of** — places where the team knows a better shape exists and hasn't gotten there yet.

This section is the antidote to the "tidy picture" failure mode. **Be honest.**

### 18. Decisions & rationale

For load-bearing decisions — the ones a rewrite would have to revisit — capture:

- **The decision**
- **The forces** (the quality attributes that drove it)
- **Alternatives considered**
- **Why this one was chosen**
- **What it costs** (what the system would be better at if a different decision had been made)

If the project has ADRs, **point at them** rather than duplicating. The architecture document is the index; ADRs are the depth. If the project has no ADRs and several load-bearing decisions are undocumented, propose creating ADRs for the top 3–5 in the **Open questions** section.

### 19. Open questions & areas of active change

- **Things you couldn't determine from the code alone** — explicit list. This is honest scope.
- **Active rewrites or migrations** — what's mid-flight, where the boundary is, how long the dual state will last
- **Decisions the team is currently debating** — if you can tell from the code (legacy module alongside new module; two implementations of the same thing; a `// TODO: remove after migration X` cluster)
- **Drift from the documented architecture** — if you found places where the code already disagrees with what this document says is intended

## Step 3 — Pick the right level of detail

A common failure: too much detail in some sections, not enough in others.

Calibrate per section:

- **Sections that exist as reference** (module map, integrations, env vars) — exhaustive, table-style, every entry has a file path.
- **Sections that exist for understanding** (high-level architecture, data flows, gotchas) — selective. Pick the things that, once understood, let the reader figure out the rest. **Better to deeply explain 3 flows than shallowly list 30.**
- **Sections that exist for honesty** (gotchas, decisions, open questions) — concise but specific. One sentence per item is often enough; padding hides the signal.

Length target for the whole document: **1,000–2,500 lines for a serious system; 300–800 for a small one.** If you're heading past 3,000, you're either documenting things that should be in subsidiary docs (move them to `docs/<subsystem>.md` and link) or padding.

## Step 4 — Prefer diagrams where they earn their keep

Mermaid is supported in most rendering pipelines. Use it where a diagram pays for itself:

- **`graph TD` / `flowchart`** for component / dependency diagrams (Step 5)
- **`erDiagram`** for the data model (Step 8)
- **`sequenceDiagram`** for flows that cross services or have meaningful async steps (Step 7)
- **`stateDiagram-v2`** for entities with non-trivial lifecycle (orders, jobs, subscriptions)
- **`C4Context` / `C4Container`** if the project already uses C4

Don't add a diagram per section out of habit. A bad diagram (too many nodes, unclear edges, every box at a different abstraction level) is worse than no diagram. **Each diagram has one job; if you can't say what it is in one sentence, redo it or delete it.**

ASCII diagrams are fine for tiny structures (3–6 boxes) where Mermaid would be overkill.

## Step 5 — Generate the document

Produce **one file**: `docs/ARCHITECTURE.md`. Output the file content only — no preamble, no commentary, no "here is the document" wrapper.

For the file itself:

- Use the section structure from Step 2, **adapted to this system**. Don't include sections that don't apply; do add ones the system needs.
- Reference real file paths everywhere a claim is made. Tables with file paths are more useful than prose.
- Be specific. "The auth middleware lives in `src/middleware/auth.ts:14`" beats "auth is handled in middleware."
- **Never invent**. If you don't know, say so in **Open questions** or omit and note the gap.
- **Distinguish fact from inference.** Code says X; design intent appears to be Y; that's an inference. Use language like "appears to" or "inferred from" when you're reading the system's intent rather than its statement.
- Keep every section answering its own question. Don't bleed one section's content into another (auth in **Cross-cutting**; security implications of auth in **Security model**; same fact, two angles).
- **Don't paste large code blocks**. If a snippet is essential, keep it short and link to the file. The architecture document is index and explanation, not source code mirror.
- Use callouts (blockquote with a label) for things that matter at a glance: footguns, deprecated patterns, ownership boundaries.
- Put a **last-updated date** at the top, plus a one-line summary of what changed if this is a revision (architecture documents must be living; an out-of-date one is a hazard).

## Constraints

- **Honesty over tidiness.** If the system has accidents, vestigial parts, or known anti-patterns, the document says so. A polished document that omits the messy parts trains people to ignore the messy parts.
- **Don't aspire.** Don't describe how the system _should_ work; describe how it does. If there's a planned redesign, mention it in **Open questions / Active change**, not in the body.
- **Specific over abstract.** Every claim grounded in file paths. "Layered architecture" alone is not a finding; "layered architecture: HTTP layer in `src/api/`, service layer in `src/services/`, data layer in `src/repos/`, with a strict no-cross-layer-import rule enforced by `eslint-plugin-boundaries`" is.
- **Don't restate code.** A list of every file in `src/` is not architecture. The architecture is what holds those files together and how they're constrained.
- **Don't pad with "best-practice" prose.** Every line earns its place by saying something specific about _this_ system.
- **Don't try to cover everything.** If the system is large, scope explicitly and say what's deferred. A focused document is more useful than a vague comprehensive one.
- **Match the project's vocabulary.** Use the names the team uses (their domain language, their service names, their internal jargon). A document in academic-textbook English about a system everyone calls something else is unusable.
- **Date it. Sign it.** A note at the top — date, author or "generated with assistance from <model> on <date>", commit SHA the analysis was based on — lets future readers know how stale it is.

## What the user provides

- **Scope** — whole repo (default) or a specific subsystem
- **Optional context that sharpens the document**:
  - Audience (new engineer, security review, AI context, external auditor) — affects emphasis
  - Sections to deepen or skip
  - Known load-bearing parts the user wants captured deliberately
  - Known anti-patterns or vestigial areas the user wants surfaced (or, conversely, areas the user considers settled and doesn't want re-litigated)
  - Existing docs to incorporate or supersede
  - Length budget if there's a strong preference

If only "write architecture docs" is provided with no scope cue, default to whole-repo at the depth the codebase supports — but state explicitly in the document's introduction what was treated as deep vs shallow, so the reader knows where the gaps are.
