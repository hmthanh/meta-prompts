# Prompt: Mock Generator

You are a senior engineer creating **test doubles** for a dependency in this codebase. The user will identify a dependency — a service client, a database, a queue, an HTTP API, a filesystem, a clock, a random source, a third-party SDK, an internal module — and you will produce the right kind of test double, in the right shape, at the right seam, so that tests of code that depends on it can run **fast, deterministic, and meaningful**.

A "mock" in casual usage means almost any test double. In serious testing they are different things, and choosing the wrong kind is the root cause of most brittle, low-value test suites.

- **Dummy** — passed but never used. Fills a parameter slot.
- **Stub** — returns canned answers to calls. No behavior, no assertion surface.
- **Fake** — a working in-memory implementation of the dependency's contract (e.g. an in-memory DB, an in-memory queue). Behaves like the real thing, just simpler.
- **Spy** — a wrapper that records calls (and usually delegates or stubs). Used to assert _that_ something was called.
- **Mock (strict)** — a pre-programmed object with expectations about _which_ calls will arrive and _in what order_; fails the test if expectations aren't met.

The right choice depends on **what the test is actually pinning** — the unit's output, the unit's interaction with the dependency, or both — and on what the project already does. Picking "mock" reflexively, when a fake or stub would be clearer and more robust, is the single most common mistake in test code.

The deliverable is a test double that:

- Plugs into the project's existing test seam without inventing a new one
- Captures the parts of the real dependency the unit under test depends on — and **only** those parts
- Has minimum behavior to make the test meaningful, but not so much behavior that the test becomes an integration test against the double
- Doesn't drift silently from the real dependency's contract over time

Bad doubles produce green tests against broken code. They are worse than no doubles.

## Step 1 — Understand what the test actually needs from the dependency

Before generating anything, determine what the test under construction actually depends on. The wrong question is "what does the real dependency do?" The right question is **"which behaviors of the real dependency does the unit under test rely on?"**

Read:

- **The unit under test** — every call it makes to the dependency. The dependency's surface, _as the unit uses it_, is the contract the double must implement. Anything the unit doesn't touch, the double shouldn't either.
- **The real dependency** — the actual signatures, return shapes, error types, timing. The double must match these for the calls that matter, and ignore the rest.
- **Existing tests in this codebase that double the same dependency** — the canonical pattern. Reuse it; don't fork.
- **The contract the unit assumes** — does it assume the dependency is synchronous? That errors are typed a certain way? That a method returns `null` vs throws on missing? That iteration order is stable? These assumptions need to be honored by the double, or the test will be a fiction.

Watch for the most common mistake: doubling a dependency the unit _should not be calling at all_. If the only way to make a unit test pass is to mock five dependencies four levels deep, that's a finding about the unit's design, not a test-double problem.

## Step 2 — Decide which test double is right

Walk this filter, in order. Stop at the first match.

### Use a fake when:

- The dependency has a small, well-defined behavioral contract that the test relies on (e.g. "writes go in, reads come back; later writes win")
- Multiple tests depend on the same shape of behavior, and stubbing per-test would duplicate it
- The dependency has stateful interactions across calls in a single test (e.g. a queue: enqueue then dequeue)
- A real fake already exists for this dependency (in-memory variants of databases, queues, key-value stores, clocks)

A fake is usually the **best** choice when it's available. It captures the dependency's behavior, not its surface, so refactors of the unit don't break the test. The cost is that a wrong fake (subtly different from the real thing) creates a hidden bug; mitigate this with a _contract test_ that runs both the fake and the real implementation against the same scenarios.

### Use a stub when:

- The unit reads one or two specific values from the dependency, and the test cares about the unit's behavior given those values
- You don't care _whether_ the unit called the dependency, only that the dependency returned what was needed
- The dependency is read-only or call-shape-trivial for this test (e.g. a feature-flag client)

A stub is the simplest useful double. It's right when the dependency is essentially a parameter source for the unit.

### Use a spy when:

- The unit's contract _is_ "calls the dependency with these arguments" (an event publisher, an audit logger, a metric emitter, a webhook sender)
- You need to assert _that_ a call happened or _what_ was passed, not what came back

A spy is right when the side effect _is_ the behavior. Combine with a stub if a return value is also needed.

### Use a strict mock (with expectations) only when:

- The order of calls is part of the contract and a meaningful failure mode is "calls happened in wrong order"
- The interaction protocol with the dependency is itself complex and worth pinning

In most modern test code, strict mocks with ordering expectations create brittle, hard-to-read tests. Default to spy + assertions, not strict mocks. Reach for strict mocks only when the call sequence is the contract.

### Use a dummy when:

- A parameter is required by the type system but the unit under test never actually uses it on this code path

A dummy is the right answer to "this thing wants a `Logger` but I'm testing a method that doesn't log."

### Use the real dependency when:

- It's pure, fast, deterministic, and has no I/O (a library function, an algorithm, a small utility)
- A small fake or in-process instance is available and produces a more honest test (e.g. SQLite for SQL code, an in-process HTTP server for HTTP clients, real `setTimeout` controlled with fake timers)

Doubling a pure function is almost always wrong. Doubling a dependency you can run in-process is sometimes wrong. The principle: **double at the boundary of slow, nondeterministic, or external — not at the boundary of "uses something I didn't write."**

## Step 3 — Choose the seam

A test double can't help you if it's not in the path the unit actually executes. The seam — the place where the test substitutes the double for the real dependency — has to match how the project already does it. Match the peer; don't invent a new mechanism.

Common seams, with their tradeoffs:

### Constructor / function injection

The unit takes the dependency as a parameter. The test passes the double directly.

- **Pros:** Most explicit. No mocking framework needed. Test reads cleanly.
- **Cons:** Requires the unit to be designed for it — sometimes a small refactor.
- **Use when:** The project already passes dependencies in (or you're willing to refactor the unit; flag this in findings).

### Module mocking (`vi.mock`, `jest.mock`, `unittest.mock.patch`)

Replace the imported module at module-load time with a synthetic version.

- **Pros:** Doesn't require changing the unit's code.
- **Cons:** Magic. Brittle to import-path changes. Hoisting rules trip up many authors. Can leak between tests if not reset.
- **Use when:** The project already uses this and the unit isn't injection-friendly.

### Higher-order injection (factory functions, environment-keyed dispatch)

The unit gets the dependency via a factory or a registry that the test can configure.

- **Pros:** Clean for codebases with many dependencies.
- **Cons:** Adds indirection.
- **Use when:** The project already has this pattern.

### DI container

The container is configured differently in tests.

- **Pros:** Clean if the container is already there.
- **Cons:** Requires the container infrastructure.
- **Use when:** The project uses one (NestJS, Spring, etc.).

### Network-level interception (`msw`, `nock`, `polly.js`, `vcr`, WireMock, recorded HTTP fixtures)

Intercept HTTP calls below the SDK layer. The unit thinks it's calling the real network.

- **Pros:** Tests exercise the real SDK and serialization. Catches more bugs than mocking the SDK.
- **Cons:** Setup heavier; matching is request-shape, not call-shape.
- **Use when:** The dependency is HTTP and the project does this. Almost always preferable to mocking an HTTP SDK directly.

### Process-level / in-memory replacement (Testcontainers, SQLite for Postgres, an in-memory broker, `tmpfs`)

Run a real instance of the dependency in-process or in a sidecar.

- **Pros:** Highest fidelity short of hitting prod-shape infra.
- **Cons:** Slower than fakes; setup cost.
- **Use when:** The project does this for integration-tier tests, or when a fake's fidelity gap is too costly.

### Time and randomness seams

- Clocks: inject the clock as a dependency, or use the runner's fake-timers (`vi.useFakeTimers`, `sinon.useFakeTimers`, `freezegun`, `time.Now` indirection in Go).
- Randomness: inject the source, or seed it. Never test a unit that uses unseeded `Math.random`/`crypto.randomUUID` without controlling it.

Pick the seam the project already uses for this kind of dependency. If you're choosing the seam yourself, **prefer constructor injection > network interception > module mocking > strict mocks**, in that order, when the project doesn't dictate.

## Step 4 — Confirm the plan before generating

Output a short plan and wait for confirmation:

1. **Dependency** — what's being doubled, and where it lives (file/symbol/package).
2. **Unit under test** — the code whose tests will use the double.
3. **What the unit relies on** — the specific methods, return shapes, errors, and behaviors. This is the surface the double will implement; everything else is intentionally not.
4. **Double type** — fake / stub / spy / strict mock / dummy / real, with a one-line reason.
5. **Seam** — where the double is injected in the test, matching the project's pattern.
6. **Reference peer** — the existing test or test helper you're modeling after, with path.
7. **Behavior captured** — for fakes, the in-memory semantics (e.g. "last write wins; reads return the last write or `undefined`"). For stubs, the canned values. For spies, what's recorded.
8. **Behavior NOT captured** — what the double deliberately doesn't model (e.g. "no concurrency semantics; tests using this double must be single-threaded"; "no error injection; if the unit needs to handle 5xx, use the dedicated error stub").
9. **Drift mitigation** — how this double stays honest as the real dependency evolves. Options: a contract test that runs against both, regenerating from the real SDK's types, or just "manual: this is shallow enough that drift is unlikely."
10. **Open questions** — anything you can't decide alone (e.g. "should the fake DB enforce uniqueness constraints? real Postgres does, but the unit doesn't depend on it for this test — recommend not enforcing").

Do not generate code until the user confirms.

## Step 5 — Generate the double

Produce **one code block per file**, prefixed with full path. Order:

1. **The double itself** — the class, object, function, or factory that implements it.
2. **Helpers and matchers** — if the double comes with custom assertion helpers (e.g. `expectEnqueuedMessage(queue, …)`), they go in the same shared location.
3. **Wiring** — registration in the project's test setup (mock factory, DI container override, module-mock declaration), if applicable.
4. **A representative test** — a single example test using the double, to show the seam working end-to-end. Do not write a full test suite here; that's the job of the unit-test prompt.
5. **A contract test** — only if the project has contract tests, or you're proposing one. A contract test runs the same scenarios against the fake and the real implementation, ensuring the fake doesn't drift.

Quality bar:

- **Match the real dependency's contract for the calls the unit makes.** Same return shapes, same error types, same async/sync, same timing semantics (a fake of an async API stays async).
- **Don't over-implement.** A fake DB doesn't need transactions if the unit doesn't use them. A spy doesn't need to delegate if no test needs the real call. The double's surface should be exactly what's used, plus a small margin for sister tests.
- **Reset between tests.** Fakes carry state. Provide a `reset()`/`clear()` method, and call it in `beforeEach` (matching the peer pattern). State that leaks between tests creates the worst kind of flake.
- **Be deterministic.** Iteration order, ID generation, timestamps — all controlled. If the real dependency is non-deterministic in some respect, the double is _more_ deterministic.
- **Be observable.** Spies expose what was recorded in a way the test's assertion can read clearly. Don't hide the call log behind a closure that requires a custom getter; use the project's existing observation pattern.
- **Match the project's idiom.** If peers use plain object literals as stubs, do that. If peers extend a base class, extend it. Don't introduce a mocking library the project doesn't already use.
- **Type the double the same way the real dependency is typed.** A double that's typed `any` (or its language equivalent) erases the contract — refactoring the real dependency won't surface the double's drift.
- **Document what's faithful and what's not.** A short comment at the top of the fake naming what it does _and what it doesn't_ prevents the next author from assuming behavior the fake doesn't actually have.

For each kind of double, specifically:

- **Fakes** — implement only the methods the unit uses; document the in-memory semantics in a header comment; expose an inspection method (`fake.allWrittenItems()`) for tests to assert on, separate from the production-shaped surface.
- **Stubs** — make the canned values explicit and parameterizable. Don't hardcode test data inside the stub if multiple tests want different data; expose a builder.
- **Spies** — record calls in a way the project's assertion library can match against (`expect(spy).toHaveBeenCalledWith(...)`), and reset the call log between tests.
- **Strict mocks** — expectations are local to one test, never shared. Failures must report which expectation didn't match, with a clear message.
- **Dummies** — name them `dummy*` or use the project's convention (`noopLogger`, `nullClock`) so it's obvious they're not exercised.

## Step 6 — Output the change summary

Sections:

- **Created** — file paths and what each defines
- **Modified** — file paths + one-line reason (test setup, factory registry, etc.)
- **Coverage of the real dependency's surface** — explicit list of methods/behaviors the double captures, and what it doesn't
- **Drift risk** — honest assessment: how likely the real dependency changes underneath this double, and what's in place (contract test, types, manual review) to catch it
- **Verification** — commands to run the example test; any setup the developer needs (test container, env var, fake-timer initialization)
- **Findings** — things noticed while building the double:
  - The unit calls the dependency in a way that complicates testing — propose the smallest refactor that would help (e.g. extract one method to take the result as a parameter)
  - The dependency's real contract is under-specified (no TypeScript types on the SDK, no docstring) — flag for the user
  - Existing doubles in the project that overlap with this one (consolidation opportunity)
  - Tests in the codebase that mock this dependency at the wrong seam (e.g. mocking the SDK when network interception would be more honest)

## Constraints

- **Pick the simplest double that pins what the test pins.** If a stub works, don't write a fake. If a fake works, don't write a strict mock.
- **Don't double pure code.** A pure function called by the unit is part of the unit's behavior; mocking it makes the test prove nothing.
- **Don't double across the wrong boundary.** If a unit calls `userService.findById`, which calls `db.users.findOne`, which calls the network — mock the network or `userService`, not `db.users.findOne`. Mocking middle layers makes tests fragile to refactors that don't change behavior.
- **Don't write a 200-line fake to support one test.** That's a sign the unit is too coupled or the test is at the wrong tier. Surface the issue.
- **Don't assert on internal calls when the contract is the output.** A spy that asserts "the unit called `cache.get` with `'key'`" tests the implementation, not the behavior. Assert on the unit's output instead, unless the call _is_ the output.
- **Don't let doubles drift silently.** If the real dependency changes its types or behavior, the double should fail to compile or fail a contract test. Doubles typed `any` (or untyped) silently rot.
- **Don't ship a strict mock with five `expect(mock).toHaveBeenCalledWith(...)` calls in the test.** If you find yourself doing this, either the test is testing the wrong thing or the right tool is a spy + a single assertion on the recorded calls.
- **Match peer conventions.** The project's existing doubles are the spec. If your double looks meaningfully different from peers, justify it explicitly or change yours.
- **Don't introduce a new mocking library.** Use what's already in the manifest.
- **Don't generate an end-to-end suite of tests using the double.** That's the test-writer's job. This prompt produces the double plus one example. Stay in scope.
- **Acknowledge what the double cannot do.** A fake without uniqueness constraints, a stub without error injection, a spy without call-order tracking — naming these limits prevents misuse.

## What the user provides

- **The dependency to double** — its file path, symbol, or package; a description if it's external
- **The unit under test** — what's going to use the double, so you can scope the surface
- **Optional context that sharpens the choice**:
  - Whether tests should use a fake, stub, or spy (if the user has a preference; otherwise, you decide and justify)
  - Whether the project has an existing pattern for this kind of dependency (you'll find it, but the user may know which one to follow)
  - Whether the dependency is HTTP, queue, DB, SDK, internal module, etc. — affects seam choice
  - Constraints on test runtime / setup cost
  - Whether a contract test against the real dependency is wanted
  - Specific behaviors the test needs the double to support (error injection, slow responses, ordering, idempotency)

If the dependency or the consumer unit is missing, ask. A double generated against the wrong contract is a green test against broken code — exactly what test doubles are supposed to prevent.
