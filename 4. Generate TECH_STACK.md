# Prompt: Generate TECH_STACK.md

You are a senior engineer documenting the technology choices behind this project. I have just cloned this repository. Produce a single file `docs/TECH_STACK.md` that gives a **fast, accurate, opinion-aware** picture of every technology in play — not just _what_ is used, but _why it likely was chosen_ and _what it implies_ for anyone working on the system.

This is distinct from `DEPENDENCIES.md` (which catalogs packages) and `ARCHITECTURE.md` (which describes structure). This file answers: **"If I'm joining this project, what worldview do I need to adopt?"**

## Step 1 — Discover

Read enough of the repo to ground every claim:

- Manifests and lockfiles for every language present
- `Dockerfile`, `docker-compose.yml`, `.tool-versions`, `.nvmrc`, `.python-version`, `runtime.txt`
- CI configs (`.github/workflows`, `.gitlab-ci.yml`, `circle.yml`, etc.)
- Framework config files (`next.config.js`, `vite.config.ts`, `tsconfig.json`, `pyproject.toml`, `settings.py`, `application.yml`, etc.)
- ORM / migration directories
- Testing config (`jest.config`, `vitest.config`, `pytest.ini`, `playwright.config`, etc.)
- Linter / formatter config
- IaC files (`terraform/`, `pulumi/`, `serverless.yml`, `cdk.json`)
- README and any `docs/`

Cross-check versions from manifest **against** lockfile **against** runtime config — they sometimes disagree, and that disagreement is itself a finding.

## Step 2 — Write `docs/TECH_STACK.md` with these sections

> **Framework:** Next.js 14 (App Router)

> **Infra:** Vercel + Supabase

### 3. Languages & runtimes

For each language used:

- Version and where it's pinned
- Strictness / config posture (e.g. `strict` TS, `mypy --strict`, Go modules version, Rust edition)
- Style conventions in force (formatter, lint rules of note)
- Anything non-default worth knowing (path aliases, experimental features enabled)

### 4. Frameworks & their conventions

For each framework (web, API, ORM, UI, worker, etc.) — one short subsection:

- **Version & key features used** (e.g. "Next.js App Router with RSC, no Pages Router")
- **The opinions it imposes** on this codebase (file-based routing, dependency injection style, etc.)
- **What patterns the code follows** vs the framework's defaults
- **Escape hatches in use** (custom middleware, monkey-patches, ejected configs)

### 5. Data layer

- Database engine and version, hosted vs self-managed
- Schema management approach (migrations tool, manual SQL, ORM push)
- Where the schema lives
- Connection pooling, read replicas, multi-tenancy posture if any
- Any non-obvious choices (JSON columns, partitioning, custom types, extensions like `pgvector`, `postgis`)

### 6. Infrastructure & deployment

- Where it runs (cloud, region, platform)
- How it's built (container? serverless bundle? static export?)
- How it's deployed (CI step, manual, GitOps)
- Environments (dev / staging / prod) and how they differ
- Secrets management approach

### 7. Testing strategy

- What's tested at each tier (unit / integration / e2e / contract / load)
- Coverage expectations if visible (CI thresholds, badges)
- Fixtures / factories / test data approach
- Anything mocked at the boundary (HTTP, DB, time, randomness)

### 8. Observability & operations

- Logging library and format (structured? JSON?)
- Metrics / tracing (OpenTelemetry? vendor SDK?)
- Error reporting (Sentry, Rollbar, etc.)
- Health checks, readiness probes
- Feature flags, if any

### 9. Why this stack — likely rationale

For each **load-bearing** choice (the ones a rewrite would have to revisit), one short paragraph inferring the _why_:

- What problem it solves for this project specifically
- The realistic alternatives that were not chosen
- What it costs (learning curve, lock-in, ops burden)
- Whether the choice still looks right given the code's current shape

Be honest. If a choice looks like accidental complexity or legacy drift, say so. Mark inferences as inferences.

### 10. Version posture & freshness

- Anything pinned to an old major (and what the current major is)
- Anything tracking `latest` / floating ranges that could drift
- Mismatches between manifest, lockfile, CI image, and Dockerfile
- Deprecated APIs in use that you can spot

### 11. Constraints this stack imposes

What a contributor needs to internalize:

- "We don't use X here" rules implied by the setup
- Performance characteristics (cold starts, memory ceilings, rate limits)
- Platform limits (e.g. Vercel function size, Lambda timeout, edge runtime restrictions)
- Anything that would make a "reasonable" change actually break in this stack

### 12. Open questions

Decisions you can't justify from the code alone — the kind a tech lead should answer:

- Why X over the obvious Y?
- Is this version pin deliberate?
- Is this layer a deliberate choice or vestigial?

## Constraints

- Every version number must come from a real file. Cite the file path inline if it isn't obvious.
- Distinguish **fact** (what the code says) from **inference** (why it was chosen). Use language like "likely" / "appears to" for the latter.
- No vendor cheerleading. No "X is the industry standard" filler. Say what _this project_ does with it.
- Tables for inventories, prose for rationale. Don't pad either.
- Keep it under ~600 lines.
- Output the file content only — no preamble.
