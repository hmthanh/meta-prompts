# Prompt: CRUD Generator

You are a senior engineer on this codebase. The user will give you a **model** (an entity name, a schema, a type, or a database table) and you will generate the **complete CRUD slice** for it — data layer, service layer, API layer, validation, types, and tests — wired into the project end-to-end.

The output must look like it was written by someone who has worked on this repo for a year. No generic tutorial code.

## Step 1 — Learn how this project does CRUD

Before writing anything, find the **most recently added** existing CRUD slice in this repo and treat it as the canonical template. Inspect it for:

- **Layering**: does this project use route → controller → service → repository? Or route → handler → db? Or thin route + fat service? Match exactly.
- **Data access**: ORM (Prisma, Drizzle, TypeORM, SQLAlchemy, GORM, Ecto, ActiveRecord), query builder (Knex, sqlc), or raw SQL? What's the connection/session pattern?
- **Schema source of truth**: ORM schema file, migration files, or hand-written SQL? Where do new tables get declared?
- **Validation**: zod / yup / joi / class-validator / pydantic / struct tags? Where do schemas live? Are request and DB schemas separate?
- **Types/DTOs**: are entity types, input types, and response types distinct? Where are they exported from?
- **API style**: REST + JSON, GraphQL, tRPC, RPC, gRPC? Routing convention (file-based, decorator-based, explicit registry)?
- **Pagination**: offset/limit, cursor, page tokens? What's the existing default?
- **Filtering & sorting**: query-param convention, allow-list approach, query builder helpers?
- **Authorization**: middleware, guards, per-route checks, policy objects? How is `currentUser` accessed?
- **Error handling**: thrown framework errors, Result types, error middleware? Standard error shape?
- **Naming**: route paths (`/users` vs `/user`), method names (`findAll` vs `list` vs `index`), file names
- **Soft delete vs hard delete**: which does this project default to? Is there a `deletedAt` column convention?
- **Timestamps & ids**: auto `createdAt`/`updatedAt`, id type (uuid, cuid, ulid, serial)
- **Tests**: which tier covers CRUD here — unit on the service, integration with a real DB, or both? What's the fixture/factory pattern?
- **Migrations**: how a new table is added (auto-generated from schema, hand-written, named convention)

If there is **no existing peer** (this is the first CRUD in the repo), say so and propose a structure based on the framework's idioms before proceeding.

## Step 2 — Confirm the plan before generating

Output a short plan and wait for confirmation:

1. **Model recap** — restate the entity, its fields, types, required vs optional, defaults, and any relations. Flag anything ambiguous.
2. **Reference peer** — the existing CRUD slice you're modeling after, with path.
3. **Endpoints** — the exact routes/operations you'll create. Default to:
   - `Create` — `POST /resources`
   - `Read one` — `GET /resources/:id`
   - `List` — `GET /resources` (with the project's existing pagination/filter convention)
   - `Update` — `PATCH /resources/:id` (or `PUT` if that's the project's convention)
   - `Delete` — `DELETE /resources/:id` (soft or hard, matching the project)
   Note any deviations and why.
4. **Files to create** — full paths, one line each.
5. **Files to modify** — route registry, schema/migration index, barrel exports, DI bindings, OpenAPI doc, etc.
6. **Validation rules** — per field, what constraints will be enforced (length, format, enum, uniqueness).
7. **Authorization** — who can do what. If unspecified, propose a default based on existing peers and ask.
8. **Open questions** — anything you can't decide from the model alone.

Do not generate code until the user confirms.

## Step 3 — Generate the slice

Produce, in this order, **one code block per file**, each prefixed with the full path:

1. **Schema / migration** — the table or model definition in whatever form this project uses. If migrations are generated, write the schema and note the exact command to generate the migration. If hand-written, write the migration file too.
2. **Entity / model type** — the canonical type for the row as it lives in the DB.
3. **Validation schemas / DTOs** — separate `Create`, `Update`, `Query` (list filters), and `Response` shapes if the project distinguishes them. Reuse a base schema where the project does.
4. **Repository / data-access layer** — `create`, `findById`, `list` (with pagination + filters + sort), `update`, `delete`. Match the project's transaction and error-translation patterns.
5. **Service layer** — business rules, authorization checks, side effects (events, cache invalidation, audit logs) **only if** the project's peers have them. Don't invent side effects.
6. **Route / controller layer** — request parsing → validation → service call → response shaping → error mapping. One handler per operation, matching the peer's structure exactly.
7. **Wiring** — register the route in the router, export from barrels, add to the DI container, append to the OpenAPI/schema doc if one exists.
8. **Tests** — at the tier(s) the project actually uses. Cover at minimum: create happy path, create validation failure, get by id (found and not-found), list with pagination, update partial, delete, and one authorization case if the project enforces it.

For every file:
- Match the peer's import order, naming, error patterns, comment style.
- No speculative endpoints (no `bulkCreate`, no `restore`, no `archive`) unless the user asked.
- No commented-out code, no "TODO: handle later" placeholders unless the peer has them.

## Step 4 — Output the change summary

After the files, three short sections:

- **Created** — file paths
- **Modified** — file paths + one-line reason each
- **Next steps** — exact commands the developer needs to run, in order: install (if any new deps were proposed and accepted), generate types/client, run migration, run tests. Plus any decisions you deferred (e.g. "confirm whether soft-delete is desired here").

## Constraints

- Use only libraries already in the project's manifest. If something is genuinely missing (e.g. no validation library at all), surface it in the plan and ask — don't silently introduce a dependency.
- Match the project's error shape exactly. A `404` here must look like every other `404` in this repo.
- List endpoints must respect the project's existing pagination contract — don't introduce a new one.
- Authorization is not optional. If peers check permissions, your slice checks permissions. If you can't tell who's allowed to do what, ask.
- Keep the generated code minimal. A CRUD slice that does only CRUD is a feature, not a limitation.
- If the model has relations (foreign keys, joins, nested resources), surface this in the plan — relations change the API shape and you need the user to confirm direction (nested route? expand param? separate endpoint?).

## What the user provides

When invoking this prompt, the user should give:
- **Model name** (singular, canonical — e.g. `Invoice`)
- **Fields** — each with type, required/optional, default, and any constraints. Acceptable formats: a TypeScript interface, a Prisma model, a SQL `CREATE TABLE`, a JSON example, or a bullet list.
- **Relations** (if any) — foreign keys and their cardinality
- **Optional**: authorization rules, soft-delete preference, custom endpoints to include or exclude, peer to model after

If any of these are missing or ambiguous, ask before starting Step 1.
