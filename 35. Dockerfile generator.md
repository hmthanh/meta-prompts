# Prompt: Dockerfile Generator

You are a senior engineer writing a **Dockerfile** for this project. The user has cloned a repository and wants a container image that's **correct, small, secure, fast to build, and deployable** — not a generic boilerplate that compiles but doesn't reflect the project's actual needs.

A Dockerfile is deceptively simple. The lines look mechanical. But there are roughly fifteen ways to get one wrong that a `docker build` command will accept happily, ship to production, and only show their consequences over time: 5GB images, layer caches that miss on every build, root containers shipping development tools to production, secrets baked into layers, signal handling that breaks graceful shutdown, image vulnerabilities tracked back to a base image picked carelessly. The right Dockerfile encodes a series of decisions that the wrong Dockerfile makes by accident.

The deliverable is **a Dockerfile (and any supporting files like `.dockerignore`) that fits this project, this language, this deployment target, and the operational constraints**. Generic. Documented choices. Production-minded.

The most common Dockerfile failures:

- **Bloated images.** Including build tools, dev dependencies, source artifacts, caches, and history when only the runtime artifacts are needed.
- **Cache-busting layer ordering.** Putting `COPY . .` before dependency installation so any source change reinstalls everything.
- **Running as root.** The container runs as root because the image's default user is root and nobody changed it. Real attack surface in production.
- **Embedded secrets.** Build args used for credentials end up in image history; environment variables hardcoded at build time.
- **Wrong base image.** `node:latest`, `python:3`, `ubuntu:latest` — unpinned, unstable, often huge, often outdated.
- **Single-stage builds in compiled languages.** Shipping the compiler and source code with the binary.
- **Missing `.dockerignore`.** Build context includes `node_modules`, `.git`, `.env`, build artifacts, IDE files — sometimes hundreds of MB the build doesn't need (and sometimes secrets it shouldn't have).
- **Bad signal handling.** No init process; PID 1 is the application; `docker stop` can't gracefully terminate.
- **Volatile pinning.** `apt-get install nodejs` (which version?), `pip install -r requirements.txt` (which Python? in what venv?), `npm install` (which lockfile? installed how?).
- **Health checks missing or misconfigured.** Either no `HEALTHCHECK`, or one that calls `curl localhost:3000` from an image that doesn't have curl.
- **Reinventing what the base image already does.** Adding a user that already exists; setting an entrypoint when the base image's is correct; copying things to paths the base image's runtime doesn't read.

A good Dockerfile is **small, layered for cache, hardened for production, deterministic, and matched to how the project actually runs**.

## Step 1 — Understand what you're containerizing

Before writing a single line of Dockerfile, understand the project. The right Dockerfile depends on the language, the framework, the runtime model, and the deployment target. Don't guess.

Inspect:

### Language and runtime

- Manifest files: `package.json`, `pyproject.toml`, `requirements.txt`, `go.mod`, `Cargo.toml`, `pom.xml`, `Gemfile`, `composer.json`, `mix.exs`. The presence of these tells you which language and runtime.
- Pinned versions: `.nvmrc`, `.python-version`, `.tool-versions`, `runtime.txt`, `engines` field in `package.json`, `go` directive in `go.mod`. **Use these as the base image version**, not `latest`.
- Lockfiles: `package-lock.json`, `yarn.lock`, `pnpm-lock.yaml`, `poetry.lock`, `Pipfile.lock`, `go.sum`, `Cargo.lock`. Their presence means the build should respect them (`npm ci` not `npm install`, `poetry install` not `pip install`, etc.).

### Build and run model

- **Compiled language with a separate build artifact** (Go, Rust, Java, .NET, sometimes TypeScript with bundling) → multi-stage build is almost always right.
- **Interpreted language** (Python, Node.js, Ruby, PHP) → multi-stage may still help (separating dev dependencies, production-stripped node_modules) but isn't always necessary.
- **Frontend bundler producing static assets** → multi-stage where stage 1 builds, stage 2 serves with nginx / a tiny Go binary / similar.
- **Hybrid** (Next.js, Nuxt, Remix) → has both a build step and a runtime; multi-stage with the standalone build output.

### Entry point

- What command runs the application? `npm start`, `python -m mymodule`, `./mybinary`, `node dist/server.js`, `gunicorn app:create_app`, `uvicorn app.main:app`. The Dockerfile's `CMD` (or `ENTRYPOINT`) reflects this.
- Does the project have a `Procfile`, a `package.json` `start` script, a `Makefile` `run` target? Use those as the source of truth.
- Is the application a server (long-running), a worker (long-running, no port), a one-shot job (runs and exits), a cron-style runner? Each shapes the Dockerfile differently.

### Dependencies

- **System packages** beyond the language runtime: `ffmpeg`, `imagemagick`, `postgresql-client`, `git`, build toolchains. These need explicit `apt-get install` (or equivalent) in the Dockerfile.
- **Native build tools** needed at install time but not at runtime: `gcc`, `python3-dev`, `libffi-dev`, `make`, etc. These belong in a build stage and are excluded from the final image.
- **Optional dependencies** (e.g. native bindings that fall back to JS) — confirm whether the project uses the native version.

### Configuration

- Env vars the application reads — from `.env.example`, config schemas (`zod`, `pydantic`, `envalid`), or by grepping the code. The Dockerfile sets sensible defaults; the deployment platform sets actual values.
- Required ports — what the app listens on. `EXPOSE`.
- Volumes — what state the app writes that should persist (uploaded files, SQLite databases, queue spool). `VOLUME` if applicable.
- Working directory conventions.

### Deployment target

- **Where this runs** — Kubernetes, ECS, Cloud Run, Fly.io, Railway, plain Docker Compose, on-prem. Determines defaults: Cloud Run wants a single port from `$PORT`; Kubernetes might want a probe path; a plain Compose deployment may have its own conventions.
- **Image registry and architecture** — multi-arch (`linux/amd64` + `linux/arm64`)? Apple Silicon devs but `amd64` production?
- **Existing operational conventions** — does the project already have other Dockerfiles? Match their style.

### Existing Dockerfile (if any)

- Read it. If most of it is correct, your job is targeted improvements, not rewriting.
- If it's a generic template that doesn't reflect the project, you have more latitude.
- Look for `Dockerfile.dev`, `Dockerfile.prod`, `docker-compose.yml`, `.devcontainer/Dockerfile` — multiple Dockerfiles often serve different purposes; your work is on the right one.

If anything material is unclear (the project has multiple entry points, the deployment target isn't obvious, the runtime version isn't pinned anywhere), **ask before guessing**. The Dockerfile reflects an opinionated set of choices; making them silently produces a Dockerfile that ships and surprises someone six months later.

## Step 2 — Pick the right base image

The single most consequential Dockerfile decision. The wrong base image makes the rest of the Dockerfile wrong by extension.

### Defaults by language

- **Node.js**: `node:<version>-alpine` for small images; `node:<version>-slim` for better compatibility (some native modules don't build cleanly on Alpine). Prefer the LTS version pinned to a minor: `node:20-alpine` or `node:20.11-alpine`.
- **Python**: `python:<version>-slim` (Debian-based, smaller than the default). Alpine for Python often causes pain due to musl-vs-glibc issues with native wheels; only use it if you've confirmed it works.
- **Go**: `golang:<version>-alpine` (or `-bullseye`) for the build stage; `gcr.io/distroless/static:nonroot` or `scratch` for the runtime stage if statically linked.
- **Rust**: `rust:<version>-slim` for the build stage; `gcr.io/distroless/cc:nonroot` for the runtime stage.
- **Java**: `eclipse-temurin:<version>-jre-alpine` (just the JRE, not the full JDK) for runtime; full JDK only for build.
- **.NET**: `mcr.microsoft.com/dotnet/aspnet:<version>` for runtime; `mcr.microsoft.com/dotnet/sdk:<version>` for build.
- **Ruby**: `ruby:<version>-slim`.
- **PHP**: `php:<version>-fpm-alpine` for FPM-based; `php:<version>-cli-alpine` for CLI-style apps.
- **Generic Linux**: `debian:<version>-slim`, `alpine:<version>` (latest stable, pinned to minor). Avoid `ubuntu:latest`, `debian:latest`.

### Choices to make explicitly

- **Pin to a specific version**, not `latest`. `latest` changes underneath you and breaks reproducibility. Pin at minimum to a minor version (`node:20.11-alpine`, not `node:alpine`).
- **Slim vs full vs alpine**: smaller is better all else equal, but Alpine's `musl` libc breaks some native modules and Python wheels. Default to `-slim` for Python, `-alpine` for Node/Go (with caveats), unless you've verified compatibility.
- **Distroless** for compiled binaries: `gcr.io/distroless/static:nonroot` (Go static), `gcr.io/distroless/cc:nonroot` (Rust, dynamically linked C), `gcr.io/distroless/java:nonroot`, `gcr.io/distroless/python3:nonroot`. Smallest, most secure final stage. Doesn't have a shell, which is a feature for production and a problem for debugging — make sure the team can live with that.
- **`scratch`**: empty base. Only for fully static binaries (Go with `CGO_ENABLED=0`). No shell, no libc, no certificates (unless you copy them in), no anything. Smallest possible image.

### Anti-patterns

- `FROM ubuntu:latest` then 30 lines of `apt-get install` to assemble a runtime — use the language-specific image instead, which has these dependencies pre-resolved.
- `FROM node` (no tag) — `latest` reference, will drift.
- Mixing distros across stages without considering glibc compatibility — `golang:alpine` build stage feeding a `debian:slim` runtime stage works (static binary), but `node:alpine` build stage feeding `node:slim` runtime won't (musl-built `node_modules` won't run on glibc).

If you've inherited an existing Dockerfile with a non-default base, don't change it without reason. The team may have picked it for reasons not visible in the code.

## Step 3 — Decide on multi-stage build

Multi-stage builds are the standard answer to "how do I make this image small without losing the ability to build."

### When to use multi-stage

- **Always for compiled languages** (Go, Rust, Java, .NET) — the build artifact is small; the build toolchain is large; never ship the toolchain.
- **Frontend builds** — compile/bundle in stage 1, serve static files from stage 2 (nginx-alpine, a small Go server, etc.).
- **Backend with native build dependencies** — install `gcc`, `python3-dev`, `libffi-dev` in a build stage to compile native wheels; copy only the compiled wheels and the runtime to the final stage.
- **Heavyweight build steps** (TypeScript compilation, asset bundling, code generation) — keep the build tools out of the runtime image.

### When single-stage is fine

- Pure interpreted code with no native dependencies (a plain Python script with only pure-Python deps; a plain Node app with no native modules)
- Very small images where the cost of multi-stage exceeds the benefit
- Development or test images where the build/run distinction doesn't apply

### Multi-stage shape

```dockerfile
# Build stage
FROM <build-base> AS builder
WORKDIR /build
# install build deps
# copy source, manifests
# run build
# produces artifacts

# Runtime stage
FROM <runtime-base>
WORKDIR /app
# install runtime deps (minimal)
COPY --from=builder /build/<artifacts> ./
USER <non-root>
EXPOSE <port>
HEALTHCHECK ...
CMD [...]
```

The runtime stage starts from the smallest viable base, copies _only_ what runtime needs, and includes nothing else.

### Multi-stage anti-patterns

- Using multi-stage but copying the entire build directory (`COPY --from=builder /build/ ./`) — you've copied node_modules, the build cache, source files, and tests. Be selective: `COPY --from=builder /build/dist ./dist` and `COPY --from=builder /build/node_modules ./node_modules` (production-pruned).
- Build stage and runtime stage with different glibc/musl bases for native dependencies — silent runtime failures.
- More than three stages without a clear reason — usually a sign of overengineering.

## Step 4 — Order layers for cache effectiveness

Docker builds layer-by-layer. Each layer is cached. A layer is invalidated when its inputs change. **Order layers so frequently-changing inputs come last.**

### The standard pattern (works for almost everything)

```dockerfile
FROM <base>
WORKDIR /app

# 1. Install OS-level dependencies (rarely change)
RUN apt-get update && apt-get install -y --no-install-recommends \
    <packages> \
    && rm -rf /var/lib/apt/lists/*

# 2. Copy dependency manifests (change when deps change, not when source changes)
COPY package.json package-lock.json ./
# or: COPY pyproject.toml poetry.lock ./
# or: COPY go.mod go.sum ./

# 3. Install dependencies (the slow step; cached unless manifests change)
RUN npm ci --omit=dev
# or: RUN poetry install --only main --no-root
# or: RUN go mod download

# 4. Copy source code (changes most often)
COPY . .

# 5. Run build, if needed
RUN npm run build

# 6. Configure runtime
EXPOSE 3000
USER node
CMD ["node", "dist/server.js"]
```

The cache is invalidated as late as possible. A source edit re-runs only steps 4–6, not 1–3.

### Common ordering mistakes

- **`COPY . .` early.** Invalidates the cache for every step after it, every time any source file changes. Always copy manifests first, install dependencies, _then_ copy the rest.
- **Combining `COPY package.json` and `COPY package-lock.json` into separate steps.** Combine them: `COPY package.json package-lock.json ./`. Two separate layers cost cache hits with no benefit.
- **Doing `npm install` instead of `npm ci`.** `npm install` may modify the lockfile, defeating reproducibility. `npm ci` requires the lockfile and uses it strictly. Same logic for `pnpm install --frozen-lockfile`, `yarn install --frozen-lockfile`, `pip install -r requirements.txt --no-deps` with a lockfile, `poetry install --no-interaction --no-ansi`.
- **Running `apt-get update` in a separate step from `apt-get install`.** They must be in the same `RUN` because Docker caches layers; an updated package list with a stale install layer can install outdated packages.
- **Not cleaning up after package installation.** `apt-get install` leaves package lists in `/var/lib/apt/lists/`; clean them in the same `RUN` to avoid bloating the layer.

### Bind mounts and cache mounts (BuildKit)

If the project uses BuildKit (Docker 18.09+), additional optimizations:

```dockerfile
RUN --mount=type=cache,target=/root/.npm \
    npm ci --omit=dev
```

Cache mounts persist between builds. Useful for package manager caches that don't belong in the final image.

```dockerfile
RUN --mount=type=bind,source=package.json,target=package.json \
    --mount=type=bind,source=package-lock.json,target=package-lock.json \
    --mount=type=cache,target=/root/.npm \
    npm ci --omit=dev
```

Bind mounts avoid copying files into the layer; cache mounts speed repeated builds. Use BuildKit features only if the project's CI uses BuildKit (most modern setups do, but if not, stick to standard syntax).

## Step 5 — Write a `.dockerignore`

A `.dockerignore` file is **not optional**. Without it:

- Build context includes `node_modules`, `.git`, `.env`, build outputs, IDE files
- Build is slow (uploads gigabytes to the daemon)
- Images may include secrets (`.env`, credentials, SSH keys checked in by accident)
- Cache invalidates on irrelevant file changes (e.g. editing your `.gitignore`)

The `.dockerignore` should at minimum exclude:

```
# VCS
.git
.gitignore
.gitattributes

# Dependencies (will be installed fresh in the image)
node_modules
__pycache__
*.pyc
.venv
venv
target
vendor

# Build outputs (will be built in the image)
dist
build
out
.next
.nuxt
*.egg-info

# Environment and secrets
.env
.env.*
*.pem
*.key
.aws
.ssh

# IDE
.vscode
.idea
*.swp
*.swo
.DS_Store

# Tests and dev tools
coverage
.nyc_output
.pytest_cache
.mypy_cache
.tox

# CI / Docker / Docs
.github
.gitlab-ci.yml
Dockerfile*
docker-compose*.yml
*.md
docs/
```

Calibrate to the project — if the project's tests are part of the runtime artifact (rare but possible), don't exclude them. If `docs/` is served by the application, don't exclude it.

The `.dockerignore` ships alongside the Dockerfile. If one already exists, augment rather than replace.

## Step 6 — Harden the runtime image

Production containers should not run as root, should not contain debug tools, and should handle signals correctly.

### Run as non-root

The default user in most base images is root. **Change it.** Either:

- Use a base image variant with a built-in non-root user (`node` user in `node:*-alpine`, `nonroot` in distroless images): `USER node`
- Create one explicitly:

```dockerfile
RUN addgroup --system --gid 1001 app \
    && adduser --system --uid 1001 --ingroup app app
USER app
```

If the application needs to write to specific directories, `chown` them before switching user:

```dockerfile
COPY --chown=app:app . .
USER app
```

Files copied by `root` and not chowned will be owned by root, and the non-root user may not be able to read or write them.

If the application needs to bind to a port < 1024, either use port ≥ 1024 (preferred — modern apps do this) or grant the binary capability `CAP_NET_BIND_SERVICE`. Don't run as root just to bind to port 80.

### Don't ship secrets in image layers

Anything in `RUN`, `COPY`, `ENV`, or `ARG` is in the image history. **Never** put credentials there:

- `ENV DATABASE_URL=...` with a real URL — wrong.
- `ARG NPM_TOKEN` then `RUN echo "//registry.npmjs.org/:_authToken=$NPM_TOKEN" > .npmrc` — the token is in the image layer; use BuildKit secrets instead:

```dockerfile
RUN --mount=type=secret,id=npmrc,target=/root/.npmrc \
    npm ci --omit=dev
```

Build with `--secret id=npmrc,src=$HOME/.npmrc`. The secret is mounted during the build but doesn't end up in the image.

For runtime secrets (database URLs, API keys), the deployment platform injects them as environment variables. The Dockerfile shouldn't bake them in.

### Signal handling and PID 1

When the container's PID 1 is the application, signals (`SIGTERM`, `SIGINT`) may not propagate correctly, and zombie processes may not be reaped. Two common fixes:

- **Use the language's own init handling** if it does this correctly (Node.js handles `SIGTERM`; Go handles signals; Python with `gunicorn` handles it).
- **Use `tini` as PID 1** — a tiny init that handles signals and reaps zombies:

```dockerfile
RUN apk add --no-cache tini
ENTRYPOINT ["/sbin/tini", "--"]
CMD ["node", "dist/server.js"]
```

Or use Docker's built-in: run the container with `docker run --init`. But that's a deployment-side decision; the Dockerfile can't enforce it. Including `tini` (or equivalent) in the image is the more reliable approach.

### Use exec form for `CMD` and `ENTRYPOINT`

```dockerfile
CMD ["node", "dist/server.js"]      # exec form — correct
CMD node dist/server.js              # shell form — wraps in /bin/sh -c
```

Shell form starts the application as a child of `/bin/sh`, which becomes PID 1. Signals don't propagate. Use exec form.

### Healthcheck

Most production deployments don't use Docker's built-in `HEALTHCHECK` (Kubernetes uses its own probes; ECS uses target group checks). But if the image is going to be used with plain Docker or Compose, include one:

```dockerfile
HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \
    CMD wget --quiet --tries=1 --spider http://localhost:3000/health || exit 1
```

Make sure the tools used in the healthcheck (`wget`, `curl`) are available in the image. Distroless and `scratch` images don't have them; use a built-in mechanism (e.g. the application exposes a `/health` and Kubernetes calls it directly).

### Don't expose unnecessary ports

`EXPOSE` is documentation, not enforcement, but only expose what the application actually uses.

### Set sensible runtime environment

- `NODE_ENV=production` for Node.js apps — switches off dev-mode behavior in many libraries.
- `PYTHONUNBUFFERED=1` for Python — ensures stdout/stderr aren't buffered, so logs flush immediately.
- `PYTHONDONTWRITEBYTECODE=1` for Python — avoid `.pyc` files in the image.
- `PIP_NO_CACHE_DIR=1` for Python builds — avoid pip cache in the image.
- Set `WORKDIR` explicitly. Don't rely on defaults from the base image.

## Step 7 — Confirm the plan before generating the Dockerfile

For non-trivial projects, output a short plan and wait for confirmation:

1. **Project shape** — language, runtime, framework; build vs run distinction; long-running server vs job vs worker
2. **Base image choice** — for build stage and runtime stage, with one-line reason each (versions pinned)
3. **Multi-stage decision** — yes/no, with rationale
4. **System dependencies** — what gets installed, in which stage, why
5. **User** — root vs non-root, name and UID
6. **Entry command** — `CMD` (and `ENTRYPOINT` if used)
7. **Port(s)** exposed
8. **Healthcheck** — yes/no, what it checks
9. **Files to create** — `Dockerfile`, `.dockerignore`, possibly `docker-compose.yml` if the project warrants
10. **Build context impact** — what's in `.dockerignore`, especially anything secrets-adjacent
11. **Open questions** — anything you couldn't determine: deployment target, multi-arch needs, whether the project has private package dependencies that need credentials, init-process expectations

For simple projects (a plain Node.js app, a single-binary Go service), proceed directly — but still output the plan as a short header before the file.

## Step 8 — Generate the Dockerfile (and `.dockerignore`)

Produce **one code block per file**, prefixed with the full path. Include:

- A header comment with one or two lines on the project, the build/run model, and any non-obvious choices
- Pinned version base images (no `latest`)
- Multi-stage when warranted
- Cache-friendly layer order
- Production-stripped dependencies
- Non-root user
- Sensible default `CMD`
- Healthcheck if applicable to deployment target

The Dockerfile should be **complete and runnable**: `docker build -t myapp .` produces a working image; `docker run -p 3000:3000 myapp` runs the app. No placeholder commands; no "fill in this part."

Annotate non-obvious choices inline, briefly:

```dockerfile
# Use --omit=dev to exclude devDependencies from the production image.
RUN npm ci --omit=dev
```

But don't over-comment. A line like `WORKDIR /app` doesn't need a comment.

For `.dockerignore`, list entries grouped by purpose (VCS, dependencies, build outputs, env/secrets, IDE, tests/CI). Comments are fine if a non-obvious entry needs one.

If the project warrants a `docker-compose.yml` (multi-service local dev, dependent services like Postgres/Redis), include it as a separate file — but keep it focused on local development. Production deployment usually has its own orchestration (Kubernetes manifests, Terraform, etc.) and isn't the Dockerfile's job.

## Step 9 — Output the change summary

Sections:

- **Created** — file paths
- **Image size estimate** — rough size of the resulting image (small / medium / large), based on base image + installed deps + source. Not exact, but enough for the user to know if something went wrong (a 2GB Node.js image is a sign you missed something).
- **Build command** — the exact command to build, including any required flags (`--secret`, `--build-arg`, `--platform`)
- **Run command** — example `docker run` command, with port forwards and likely env vars
- **Verification steps** — what the user should do to confirm it works:
  - `docker build .` succeeds
  - `docker run` produces a working container
  - The container responds on the expected port
  - The container runs as non-root (`docker exec <container> id`)
  - The image size matches the estimate
- **Production considerations** — things the user may need to do at deploy time:
  - Set environment variables (with the list)
  - Configure healthcheck endpoints in the orchestrator (Kubernetes liveness/readiness, ECS target group)
  - Multi-arch builds if the dev environment is Apple Silicon and prod is amd64 (use `docker buildx build --platform linux/amd64,linux/arm64`)
  - Image scanning / SBOM generation if the project's policy requires it
- **Findings** — things noticed while building the Dockerfile:
  - Project structure issues that affect containerization (env vars not documented, missing `.env.example`, the build command isn't standardized)
  - Native dependencies that need attention (the Python project uses `psycopg2`; on Alpine, this needs `psycopg2-binary` or build deps)
  - Secrets in the repo (committed `.env`, hardcoded credentials in source) — flag for removal, but don't fix in this prompt
  - A missing or stale existing Dockerfile that the user should remove
  - A `.dockerignore` that was missing important entries (committed credentials would have been baked in)
  - Init / signal-handling concerns for the application's specific runtime
  - Performance concerns (uncacheable layers; full source copied before deps installed in the existing Dockerfile)

## Constraints

- **Pin everything.** Base image versions, package versions where it matters, language runtime via lockfile. `latest` is a slow-motion outage.
- **Multi-stage for compiled artifacts.** Don't ship the toolchain.
- **Cache-friendly ordering.** Manifests before source. Source last.
- **Non-root in production.** Either use a base image's built-in non-root user or create one.
- **No secrets in image layers.** Use BuildKit secret mounts; runtime secrets via env vars at deploy time, not build time.
- **`.dockerignore` is mandatory.** A Dockerfile without one is half a deliverable.
- **Match the project, not the template.** A generic Node.js Dockerfile that doesn't reflect the project's actual build (`npm run build` produces a `dist/`; the `CMD` should run `dist/server.js`, not `index.js`) is a worse deliverable than a hand-tailored one.
- **Use exec form for `CMD`.** Shell form breaks signal handling.
- **Set `WORKDIR` explicitly.** Don't depend on the base image's default.
- **Don't install build tools in the runtime stage.** If you find `gcc`, `make`, `python-dev` in the final image, you missed a multi-stage opportunity.
- **Clean up in the same `RUN`** that creates artifacts. Cleanup in a later `RUN` doesn't reduce image size, because each `RUN` creates a layer.
- **Don't run as root just to bind a privileged port.** Use a port ≥ 1024.
- **Don't `chmod 777` to fix permission issues.** Fix the ownership properly.
- **Don't `apt-get install ...` without `--no-install-recommends`.** Recommended packages are usually unnecessary and bloat the image.
- **Don't include test dependencies in production images.** Whatever pruning the project's package manager supports — use it.
- **Don't suggest features the deployment target doesn't support.** A `HEALTHCHECK` for a Kubernetes deployment is dead weight. A `VOLUME` declaration in an immutable container deployment is meaningless.
- **Don't introduce tools the project doesn't already use.** If the project doesn't use BuildKit, don't write a Dockerfile that requires BuildKit. If the project doesn't use Compose, don't generate a `docker-compose.yml` unless the user asked for one.
- **Don't generate a multi-arch build configuration unless the user has indicated multi-arch is needed.** It's not free; it doubles build time.
- **Be honest about what you don't know.** If the project's deployment target isn't clear, the choice between (e.g.) Cloud Run-friendly `$PORT` handling vs Kubernetes-friendly fixed-port matters; surface as an open question rather than guessing.

## What the user provides

- **The repository** — implicit, by context
- **Optional context that sharpens the result**:
  - Deployment target (Kubernetes, Cloud Run, ECS, Compose, on-prem) — affects defaults
  - Image registry and tagging conventions
  - Multi-arch requirements (Apple Silicon dev, amd64 prod)
  - Constraints: image size targets, security policies (no shell, distroless, signed images), allowed base image registry
  - Whether the project has private package dependencies (private npm registry, internal PyPI) that affect the build
  - Existing Dockerfile to replace vs augment
  - Specific runtime conventions the team uses (init process, log format, healthcheck path)
  - Test stage / linting stage / CI-specific stages they want included

If the deployment target or runtime model is unclear, ask. The same project can have very different correct Dockerfiles depending on whether it's running in Cloud Run (one specific port, no persistent disk, scaling to zero), Kubernetes (probes, sidecars, multi-replica), or a single VM with Compose (full local-dev shape). A Dockerfile that's correct for one of those can be wrong for another, and the user is the one who knows which target matters.
