# Prompt: Generate ONBOARDING.md

You are a senior engineer pairing with a brand-new developer on their first day. I have just cloned this repository. Produce a single file `docs/ONBOARDING.md` written *to* that new developer — friendly, concrete, and pragmatic. The goal: they should be able to run the project, find their way around, and make their first small change within half a day of reading this.

This is **not** an architecture document (that lives in `ARCHITECTURE.md`). This is a guided tour. Assume the reader is a competent engineer who has *zero context* on this specific codebase.

## Step 1 — Explore the repo
Before writing, actually read:
- `README.md`, `CONTRIBUTING.md`, `CODEOWNERS`, `.github/`, `.editorconfig`
- Root manifest(s) and lockfile(s)
- `Makefile` / `package.json` scripts / `justfile` / `taskfile.yml`
- `.env.example` and any config templates
- Entry points and the top 2 directory levels
- Test directory layout
- Any `docs/` folder already present

Note anything that would surprise a newcomer (unusual scripts, required tools, non-default ports, mandatory env vars, weird naming conventions).

## Step 2 — Write `docs/ONBOARDING.md` with these sections

### 1. Welcome & what you're about to work on
2–4 sentences. What this project does, who depends on it, what "shipping a change" looks like here. Set expectations.

### 3. Get it running locally
A numbered, copy-pasteable sequence. Every command should actually work from a fresh clone.
- Required tools and exact versions (Node, Python, Go, Docker, etc.)
- How to install dependencies
- How to set up `.env` (list the *minimum* vars needed to boot, mark optional ones)
- How to start the database / services (docker-compose? local install?)
- How to run the app
- How to verify it works ("open http://localhost:XXXX, you should see…")

If any step commonly fails, add a **"If this breaks…"** callout right under it.

### 4. Run the tests
- Command for the full suite
- Command for a single file / single test
- Where tests live and the naming convention
- Anything slow, flaky, or requiring special setup

### 5. The 60-second tour of the codebase
A short walkthrough — *not* an exhaustive map. Pick the 5–8 directories a new dev will touch in their first month and explain each in one or two sentences with a real file as an example. Skip plumbing they won't open.

### 6. How a request flows through the system
One concrete example, end to end. Pick the most representative user action. Walk through it: file → file → file → response. This is the single most useful section — make it good.

### 7. Conventions you'll be expected to follow
- Code style / formatter / linter (and how to run them)
- Naming conventions (files, functions, branches, commits)
- How errors are handled
- How logging is done
- How config and secrets are accessed
- Anything the team is opinionated about

### 8. Your first contribution
Suggest a realistic starter task pattern (e.g. "add a field to the User model and surface it in the API"). Walk through *where* they'd make changes, *which* tests to update, *how* to verify. Don't write the code — point at the right files.

### 9. The map of "who owns what"
If `CODEOWNERS` exists, summarize it. Otherwise, note which directories seem to be touched by which kinds of changes. Mention any Slack channels, docs sites, or dashboards referenced in the repo.

### 10. Gotchas, footguns, and tribal knowledge
The stuff that isn't written down anywhere else but you can infer from the code:
- Files that look unused but aren't
- Tests that must be run in a specific order
- Migrations that need manual steps
- Env vars that silently change behavior
- Anywhere the code says "DO NOT TOUCH" or has a long apologetic comment

### 11. Where to go next
Pointers to: `ARCHITECTURE.md`, API docs, runbooks, dashboards, the deploy process. One line each.

### 12. Questions I couldn't answer from the code
List anything you couldn't figure out without asking a human — onboarding docs should be honest about their gaps.

## Constraints
- Write in second person ("you'll see…", "run this…"). It's a tour, not a spec.
- Every command must be runnable. Every file path must exist.
- No corporate fluff, no "leveraging synergies." A new dev's time is the scarce resource.
- If the repo is already well-documented in places, *link to those files* instead of duplicating.
- Keep it under ~600 lines. If it's longer, you're explaining too much.
- Output the file content only — no preamble.
