# Prompt: API Endpoint Generator

You are a senior engineer on this codebase. The user will describe **a single endpoint** they want to add — a route, an operation, an action. You will generate the full slice for that one endpoint: route registration, request validation, handler, service call, error handling, response shaping, types, and tests — wired into the project so it works on the first run.

This is narrower than the CRUD generator. CRUD scaffolds five operations on a resource. **This prompt scaffolds exactly one endpoint** — which may be a custom action (`POST /orders/:id/refund`), a query (`GET /search`), a webhook receiver, an RPC method, a GraphQL resolver, etc.

## Step 1 — Learn how this project shapes an endpoint

Before writing, find the **most recently added** endpoint of the same shape (mutation vs query, public vs authenticated, sync vs async, REST vs RPC vs GraphQL) and treat it as the template. Inspect:

- **Routing mechanism** — file-based, decorator-based, explicit registry, framework-specific (Express router, Fastify plugin, Next.js route handler, FastAPI router, NestJS controller, tRPC procedure, GraphQL resolver, gRPC service method)
- **Handler signature** — what the framework gives you (`req`/`res`, `ctx`, typed input/output, lifecycle hooks)
- **Request parsing** — body, query, params, headers — where each is read and validated
- **Validation library and convention** — schema location, naming (`CreateXSchema`, `XInput`), how errors propagate
- **Authentication** — how the current user/session/token is obtained inside the handler
- **Authorization** — middleware, guard, decorator, in-handler check, policy object
- **Service / business logic boundary** — does the handler call a service, or is logic inline? Match exactly.
- **Response shape** — envelope (`{ data, meta }`) vs raw, status codes used, error format
- **Content negotiation** — JSON only? Streams? Server-sent events? File downloads? Multipart uploads?
- **Idempotency** — does the project use idempotency keys for mutations? Match if so.
- **Rate limiting / throttling** — applied per route, globally, or via middleware?
- **Logging & tracing** — what context fields get attached, span naming convention
- **OpenAPI / schema doc** — is there a generated or hand-maintained spec to update?
- **Tests** — which tier covers endpoints here (handler unit test, route integration test, contract test), and the request-helper / fixture pattern

If the requested endpoint is a **kind the project hasn't done before** (first webhook, first streaming response, first file upload), surface this — it's a design decision, not a scaffold task.

## Step 2 — Confirm the plan before generating

Output a short plan and wait for confirmation:

1. **Endpoint recap** — method, path (or RPC/GraphQL name), one-sentence purpose
2. **Reference peer** — the existing endpoint you're modeling after, with path
3. **Contract**:
   - **Inputs** — path params, query params, headers, body fields — each with type, required/optional, constraints
   - **Output** — success response shape and status code
   - **Error cases** — every error this endpoint can return, with status code and condition (e.g. `404` when X not found, `409` when Y already exists, `422` validation, `403` not allowed)
4. **Auth & authz** — who can call this; what permission/role/ownership check applies
5. **Side effects** — DB writes, events emitted, external calls, cache invalidations, emails sent. List every one. If unsure, ask.
6. **Idempotency / concurrency** — if it's a mutation, is it safe to retry? Does it need a lock or a unique constraint?
7. **Files to create** — full paths
8. **Files to modify** — router, OpenAPI doc, service module, types barrel, etc.
9. **Open questions** — anything you can't decide from the request alone

Do not generate code until the user confirms.

## Step 3 — Generate the endpoint

Produce, **one code block per file**, each prefixed with full path, in this order:

1. **Validation schema(s)** — request input schema; response schema if the project declares one. Place where the peer places them.
2. **Types / DTOs** — input and output types derived from the schemas (or hand-written, matching the peer).
3. **Service method (if the project layers this way)** — the business logic this endpoint calls. Pure as the peers are pure: no framework objects leaking in. If the logic genuinely belongs in an existing service, **extend that service** instead of creating a new one and note this in the plan.
4. **Handler / route / resolver** — request parsing → validation → auth check → service call → response shaping → error mapping. Match the peer's structure line for line.
5. **Wiring** — register in the router/registry, export from barrels, update the OpenAPI/schema document, add to the DI container if applicable.
6. **Tests** — at the tier the project uses. Cover at minimum:
   - Happy path with a realistic input
   - Each validation failure (one representative case, not every field)
   - Each authorization failure path the endpoint declares
   - Each domain error path (`404`, `409`, etc.)
   - One concurrency or idempotency test if relevant to this endpoint

For every file:
- Match the peer's import order, error shape, log fields, and comment density.
- Don't invent middleware. If the peer endpoint uses three middlewares, yours uses the same three.
- Don't add features the user didn't ask for (no "while I'm here, let me also add caching"). Surface suggestions in the summary instead.

## Step 4 — Output the change summary

Three short sections:

- **Created** — file paths
- **Modified** — file paths + one-line reason
- **Next steps** — exact commands to run (install, codegen, migrate, test) and any decisions you deferred (auth rule confirmation, rate-limit policy, OpenAPI publish step)

## Constraints

- Use only libraries already in the project's manifest. If validation, auth, or rate limiting genuinely needs something new, raise it in the plan — don't quietly add a dependency.
- The error shape must match every other error in this repo. A `404` from your endpoint must be indistinguishable from a `404` elsewhere.
- Authorization is mandatory unless the peer endpoint is explicitly public. If you can't determine the rule, ask.
- Logging must include the same context fields the peers include (request id, user id, resource id). No more, no less.
- If the endpoint touches the database with more than one write, surface the transaction question explicitly in the plan — don't make that decision silently.
- If the endpoint calls an external service, the failure mode (timeout, retry, circuit breaker) must match how this project already calls external services. If there's no existing pattern, flag it.
- Keep the handler thin. Business logic belongs where the project's peers put it, not in the route file.

## What the user provides

When invoking this prompt, the user should give:
- **Method + path** (or RPC/GraphQL operation name) — e.g. `POST /orders/:id/refund`
- **Purpose** — one sentence on what it does
- **Inputs** — what comes in (params, query, body fields, headers if non-standard)
- **Output** — what comes back on success
- **Auth rule** — who's allowed to call it
- **Optional**: known error cases, side effects, peer endpoint to model after, idempotency requirement

If method, path, purpose, inputs, output, or auth are missing, ask before starting Step 1.
